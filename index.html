Project: Lumi (로컬 PC RAG 비서)

문제 정의

개발자의 지식 자산(코드 스니펫, 기술 문서, PDF 매뉴얼, 개인 노트, 로그 파일)은 로컬 PC 전체에 분산되어 있습니다. 이로 인해 "저번에 본 그 PDF가 어디있지?" 또는 "이 에러 로그 패턴을 이전에 다른 프로젝트에서 본 것 같은데..."와 같이 일상에서 중요한 정보를 '잊고 놓치는' 문제가 빈번하게 발생합니다.

기존의 파일 검색(예: Windows 검색)은 키워드 기반이라 파일의 '맥락'을 이해하지 못하며, 클라우드 기반 AI 서비스(예: ChatGPT)에 로컬 파일을 업로드하는 것은 민감한 정보 유출의 위험이 있어 현실적으로 불가능합니다.

목표

내 PC의 모든 파일과 "대화"할 수 있는, 100% 로컬에서 실행되어 프라이버시를 보장하는 지능형 검색 비서 'Lumi'를 VSCode 내에 구축하여 이러한 '놓치는 부분'을 자동화하고 해결하는 것이 목표였습니다.

아키텍처: 투-프로세스(Two-Process) 하이브리드

VSCode 확장 환경(TypeScript/Node.js)과 AI/데이터 처리 생태계(Python)의 이질성을 해결하기 위해 "투-프로세스(Two-Process) 하이브리드" 아키텍처를 채택했습니다.

  • 프론트엔드 (VSCode Extension): TypeScript와 React 기반의 웹뷰(Webview) UI를 담당합니다. 사용자의 모든 상호작용을 처리하고 로컬 백엔드 API를 호출합니다.
  • 백엔드 (AI/RAG Engine): Python과 FastAPI를 기반으로 하는 로컬 REST API 서버입니다. Everything 연동, 파일 콘텐츠 파싱, Ollama LLM 호출 등 모든 무거운 연산과 RAG 파이프라인을 담당합니다.
  • 프로세스 브릿지 (Bridge): 핵심 설계입니다. VSCode 확장이 활성화될 때, PyInstaller로 패키징된 Python 백엔드(Lumi_Backend.exe)를 Node.js의 child_process.spawn을 사용해 백그라운드에서 실행시킵니다. 프론트엔드는 fetch를 통해 http://localhost:8000으로 이 로컬 서버와 통신합니다. 이 방식은 사용자에게 별도의 서버 실행을 요구하지 않는 심리스(seamless)한 경험을 제공합니다.

핵심 기능 1: 초고속 로컬 파일 인덱싱 (Everything es.exe 연동)

RAG 파이프라인의 'Retrieve' 단계에서 1초 미만의 응답 속도를 달성하기 위해 Voidtools의 Everything을 연동했습니다. Everything은 NTFS 변경 저널을 실시간으로 모니터링하여 PC 전체의 파일 목록을 거의 0에 가까운 지연 시간으로 인덱싱합니다.

복잡한 C SDK나 IPC 연동 대신, 신속한 개발("바이브 코딩")과 안정성을 위해 강력한 CLI 도구인 es.exe를 채택했습니다. Python FastAPI 백엔드는 subprocess.run을 사용해 아래와 같은 명령을 실행하고, 반환된 JSON 출력을 파싱하여 RAG에 필요한 파일 경로, 크기, 수정 날짜를 즉각적으로 확보합니다. 

es.exe -json -n 10 -a-d -size -dm "검색어"

핵심 기능 2: 100% 로컬 RAG 파이프라인

사용자 PC의 프라이버시를 완벽하게 보장하기 위해 모든 AI 연산이 로컬에서만 실행되는 RAG 파이프라인을 구축했습니다.

  1. Retrieve: (핵심 기능 1에서 설명) es.exe가 사용자 쿼리와 연관된 파일 경로 목록을 반환합니다.
  2. Parse: 백엔드가 이 경로 목록을 받아, 파일 확장자에 따라 각기 다른 순수 Python 라이브러리로 실제 콘텐츠를 추출합니다.
    • .pdf: LLM용 마크다운 추출에 최적화된 pymupdf4llm 사용.
    • .docx: python-docx를 사용하여 텍스트와 테이블 내용을 추출.
    • .txt, .md, .py, .js 등: Python 내장 open() 함수로 텍스트 직접 읽기.
  3. Generate: 추출된 텍스트(Context)와 원본 쿼리를 조합하여 RAG 프롬프트를 동적으로 구성합니다. 이 프롬프트를 Ollama가 제공하는 로컬 REST API 엔드포인트(http://localhost:11434/api/generate)로 전송합니다. Ollama는 Llama 3 8B 또는 Phi-3 같은 경량화된 로컬 LLM을 사용하여 최종 답변을 생성합니다.

회고 (배운 점 및 개선점)

배운 점

  • Node.js (child_process)와 Python (FastAPI/subprocess)이라는 두 이기종 프로세스를 결합하고, PyInstaller로 패키징하여 안정적으로 통신 및 관리하는 하이브리드 데스크톱 앱 아키텍처 설계 경험을 쌓았습니다.
  • Everything의 es.exe와 같이 강력한 네이티브 도구를 SDK 없이도 CLI 래핑(wrapping)만으로도 빠르고 효율적인 백엔드 서비스로 통합할 수 있음을 배웠습니다.
  • Ollama, PyMuPDF4LLM 등 다양한 로컬 AI 및 파싱 도구들을 조합하여 복잡한 RAG 파이프라인을 신속하게 구축하는 노하우를 습득했습니다.

개선점

  • 현재는 동기식(synchronous) 파일 I/O를 사용하는데, 향후 asyncio와 asyncio.to_thread를 도입하여 다수의 파일 콘텐츠를 파싱(Parse)하는 단계를 병렬 처리하고 RAG 파이프라인의 전체 지연 시간(latency)을 최적화하고 싶습니다.
  • 사용자의 자연어 쿼리(예: "지난 주에 본 PDF")를 es.exe의 고급 검색 구문(예: ext:pdf dm:lastweek)으로 변환하는 단계를 LLM을 통해 자동화하여 1차 검색(Retrieval)의 정확도를 높이고 싶습니다.
  • es.exe 대신 Everything SDK (C DLL)와 PyEverything 같은 Python 래퍼를 도입하여, 파일 시스템 변경을 실시간으로 감지하는 기능을 구현해 볼 것입니다.
